§00 — Intention

Pourquoi cet audit

La maquette terlab_parcelles_v4h.html a stabilisé onze stratégies d'implantation, un moteur de clipping Sutherland–Hodgman et une coupe 2D complète. Notre générateur TERLAB en production n'expose pour l'instant que quatre stratégies, des reculs par adapter et une coupe schématique. Ce document dresse, dessin par dessin, l'écart géométrique à combler pour que les planches remises en mairie soient lisibles, justes et régulières vis-à-vis du PLU.

§01 — Limites de propriété

Dessiner la parcelle : la forme qui gouverne tout

La parcelle est la donnée d'entrée non négociable. Les deux outils la reçoivent en GeoJSON WGS84 puis la projettent en mètres locaux. v4h la rend modifiable et typée par edge (voie / fond / latéral) ; notre générateur la lit depuis le cadastre et la typologie d'edge vit dans l'esquisse-canvas.

v4h — parcelle typée edge par edge

sommets éditables + classification d'edge automatique (plus grande adjacence voie → fond opposé → latéraux)

Actuel — parcelle cadastre + esquisse-canvas

sommets lus depuis API Cadastre IGN ; typage d'edge délégué à esquisse-canvas.js et utilisé par envelope-generator

// v4h — typage d'edge (extrait) classifyEdges(parc, voiePoint) { const dists = parc.edges.map(e => distToPoint(e.mid, voiePoint)); const iVoie = argmin(dists); const iFond = (iVoie + Math.floor(parc.edges.length/2)) % parc.edges.length; // reste → latéral (mitoyen si alignement parcelle voisine) }

À rapatrier côté générateur : un edge-typer déterministe (voie/fond/latéral + flag mitoyenneté) pour que reculs et stratégies lisent la même vérité que la maquette v4h.

§02 — Reculs & bandes PLU

Du recul règlementaire à la zone constructible

Le cœur géométrique. v4h calcule une bande par edge via bandPoly(parc, i, r) puis clipe toutes les bandes entre elles (Sutherland–Hodgman) pour obtenir la zone. Notre adaptiveInset produit un inset global qui néglige les différentiels voie/fond/latéral et peut générer des zones fausses sur parcelles concaves ou en drapeau.

v4h — bandes per-edge + Hodgman

buildZone(parc, reculs)

// clip = Sutherland-Hodgman
zone = parc
for (b of bands) zone = clip(zone, b.outer)

  • offset = normale entrante × r (recul edge i)
  • convexe → zoneHP direct
  • concave → zoneRaster grille 0.25m
  • coins concaves → arcs 18 segments
actuel — adaptive inset global

adaptiveInset(parc, r)

// plan-masse-engine
zone = insetPolygon(parc, recul_moy)
// puis clamp AABB

  • pas de différentiation voie / fond / latéral au niveau bande
  • miter sur angles aigus → sommets éjectés (bec, drapeau)
  • pas d'arc de transition aux coins concaves
  • Sutherland–Hodgman existe dans plan-masse-canvas.js — à remonter dans le moteur
// Recommandation de port — bandPoly() à ajouter dans footprint-helpers.js function bandPoly(pts, i, r){ const [a,b] = [pts[i], pts[(i+1)%pts.length]]; const [dx,dy] = [b.x-a.x, b.y-a.y]; const L = Math.hypot(dx,dy); const [nx,ny] = [-dy/L, dx/L]; // normale intérieure (CCW) return [a, b, {x:b.x+nx*r, y:b.y+ny*r}, {x:a.x+nx*r, y:a.y+ny*r}]; }

§05 — Coupes A / B

La coupe que la mairie veut voir

v4h dessine une coupe 2D complète : profil pente, reculs projetés, volumes R+n, cotations, profMax respecté. Notre coupe-renderer.js trace un gabarit rectangulaire sur une pente constante — insuffisant pour une note de faisabilité sur terrain réunionnais pentu. L'atlas atlas-topographie-coupes-reunion.html décrit six classes de pente (0–5 · 5–15 · 15–30 · 30–50 · 50%+ · sensible) avec des règles profMax et typologies associées (pilotis, sous-plex, vide sanitaire).

v4h — coupe 2D complète

drawCoupe() — terrain + reculs + R+n + profMax + cotes · axe = stratégie primaire

Actuel — coupe schématique N-S

coupe-renderer.js · pas de MNT, pas de reculs projetés, pas d'étage

// Vers une coupe "mairie" — pipeline cible 1. échantillonnage MNT LiDAR IGN le long de l'axe stratégique (step 0.5m) 2. projection des reculs voie/fond sur l'axe → bandes d'exclusion 3. empilement R+n avec he/hf par phase, dalles dessinées 4. contrôle profMax = f(pente) cf. atlas-topographie-coupes §00–05 5. cotes + azimut + dénivelé total + légende PLU
§06 — Prospect / gabarit / CES / CBS

Les contrôles que le PLU exige

v4h borne la profondeur via profMax (pente), et affiche CES en live. Notre moteur expose Gabarit 3D et service CBS, mais la règle de prospect H/2 (distance aux limites ≥ hauteur / 2) n'est pas vérifiée sur le bâti placé. Trois zones à consolider.

à implémenter

Prospect H/2

Vérifier, pour chaque edge de la bounding box du bâti, la distance minimale à la limite parcellaire ≥ hauteur locale / 2. Cf. atlas-math-terlab Fm.04 (à enrichir).

OK

CES

Calcul déjà en place dans plan-masse-engine.metrics(). À brancher sur chaque stratégie v4h portée.

v1 isolé

CBS (Biotope)

cbs-calculator-service.js (v1) existe, légal Saint-Pierre Ua5 uniquement, pédagogique ailleurs. À relier au bpf-bridge pour cohérence plan de masse ↔ surface biotope.

// Prospect H/2 — contrôleur à ajouter dans envelope-generator ou plan-masse-engine function checkProspectH2(bldg, parc, H) { for (const edge of parc.edges.filter(e => e.type !== 'mitoyen' && e.type !== 'voie')) { const d = minDistancePolyToEdge(bldg, edge); if (d < H/2) return { ok:false, edge:edge.i, need:H/2, got:d }; } return { ok:true }; }
§07 — Atlas frères

Ce que l'atlas-math-terlab doit absorber

L'atlas math v3 couvre 51 formules (CES, RTAA, azimut, solaire 3D, MNT, placement). Il n'aborde ni les stratégies d'implantation v4h, ni les règles profMax par classe de pente, ni la règle binaire latéral/mitoyen. L'atlas batiments-collectifs fournit les typologies (lame, plot, coursive, varangue) qui nourrissent directement les stratégies 2bars / 3bars / T-cross.
